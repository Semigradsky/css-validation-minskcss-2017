Валидация

  • Один из самых ответственных и сложных этапов создания UI — веб-формы.
  • Получение, обработка пользовательских данных и обратная связь по ним — критическая составляющая успешного опыта взаимодействия пользователя (UX) с интерфейсом (UI), которая прямым образом влияет на успешность проекта.
  • Разумные ограничения валидации на стороне клиента — улучшение UX.
  • Валидация на стороне сервера — безопасность.

Constraint Validation API

  • До HTML5 проверка форм — умножение в столбик (простыни на JS). Сложные, подверженные ошибкам, сценарии.
  • После работа с калькулятором.
  • Специальное API — проверка браузером. На входе условия, на выходе ответ.
  • Все поля в любой момент в runtime провалидированы.
  • Без JavaScript.
  • Правила перекочевали в атрибуты из скриптов.
  • Валидация полей.
  • Валидация формы (не в IE/Edge).
  • Естественные ограничения на основе семантики типа поля.
  • Рассмотрим поля, с которыми будем работать.
  • email
    • значение должно соответствовать RFC 5234 (расширенная спецификация синтаксиса Бэкуса-Наура (ABNF)
    • мягкая валидация – любой символ + "@" + любой символ
    • multiple – несколько адресов через запятую (не работает Firefox)
  • tel
    • не проверяет валидность синтаксиса, т.к. в разных странах разные форматы (цифры, буквы, прочие символы)
    • включение соответствующей клавиатуры на устройстве
  • Число с плавающей запятой (точкой) — хранится в форме мантиссы и показателя степени.
  • date, datetime, datetime-local, time, month, week
  • С датами все сложно. Сегодня не рассматриваем.

*А также datetime, datetime-local, time, month, week.

  • Пустое значение всегда проходит проверку, если не добавить атрибут required.
  • required
    • Значение должно быть не пустым
    • пробел проходит
    • В элементе ввода обязательный применяется только в сочетании с определенными значениями атрибута type.
  • minlength/maxlength
    • Количество символов (in Unicode code points) не должно превышать/быть меньше значения аттрибута.
  • min/max
    • Значение поля должно быть больше/меньше или равно значению аттрибута.
    • Применимо к дате.
  • step
    • Пока в аттрибут step не установлен любой литерал, значение может быть min + любое число, кратное шагу.
    • Проверка числа на кратность допустимому шагу.
  • pattern
    • Значение должно соответствовать шаблону.
    • Регулярка Perl-подобная, с синтаксисом JavaScript, за исключением того, что атрибут pattern должен соответствовать всему значению, а не только подмножеству (except).

Constraint Validation API methods

  • checkValidity() → invalid event
  • setCustomValidity(message) → return value

Constraint validation API properties

    https://developer.mozilla.org/en-US/docs/Learn/HTML/Forms/Form_validation#Constraint_validation_API_methods
  • Метод checkValidity()
    • проверяет имеет ли элемент ограничения и соответствует ли он им
    • не соответствует — запускает событие invalid и возвращает false, соответствует — invalid и true
    • Валидность сабмитящегося элемента проверяется перед сабмитом формы или после вызова checkValidity () элемента или его формы.
  • Метод setCustomValidity(message)
    • JS-установка кастомного сообщения об ошибке на снове вычисления типа ошибки.
    • получает строкой, не пустая - ошибка, вывод сообщения; пустая - валидно, сброс сообщения об ошибке.
